# AGENTS.md - 智慧通勤综合服务平台项目约束说明

## 1. 项目概述

智慧通勤综合服务平台面向校园或园区通勤出行场景，主要围绕班车出行、共享电动车、司机服务和后台管理展开。

系统目标是为用户提供便捷的通勤服务，为司机提供清晰的任务与提醒，为管理员提供统一的运营管理入口。

当前项目仍处于需求整理和方案设计阶段，本文档暂不描述具体技术框架、开发命令、数据库状态、接口实现、测试流程和部署方案，仅用于约定项目目录、模块职责、文档组织和基本命名规则。

------

## 2. 项目目录结构

项目采用多端分层结构，一级目录固定如下：

```text
docs/       项目文档
driver/     司机小程序
wxapp/      用户小程序
fronted/    管理后台 Web
backend/    后端服务
```

各目录职责必须清晰，不能混放页面、文档或服务逻辑。

## 8. 功能归属说明

为避免多端职责混乱，功能归属按以下方式划分。

### 8.1 用户侧功能

用户侧功能放在 `wxapp/`，包括用户能看到和操作的页面，例如线路、车辆、扫码、支付、下车点、优惠券、共享电动车考试和资格查看。

### 8.2 司机侧功能

司机侧功能放在 `driver/`，包括司机任务、车辆绑定、到站确认、呼叫提醒、热点提醒和异常上报。

### 8.3 管理侧功能

管理侧功能放在 `fronted/`，包括后台管理、审核、统计、调度、二维码、优惠券、考试题库、实操审核和资格管理。

### 8.4 服务侧功能

公共数据、业务规则、接口服务和核心判断放在 `backend/`。

### 8.5 文档说明

所有需求、流程和说明放在 `docs/`。

## 9. 命名约定

### 9.1 目录命名

- 一级目录使用固定名称：`docs/`、`driver/`、`wxapp/`、`fronted/`、`backend/`。
- 业务目录建议使用英文小写。
- 多个单词建议使用 kebab-case。
- 不建议使用中文目录名。
- 不建议使用无意义名称。

### 9.2 文件命名

推荐：

```text
route-map
seat-status
dropoff-wakeup
ebike-license
coupon-list
driver-task
dispatch-map
```

不推荐：

```text
test
demo
page1
aaa
temp
new
old
```

### 9.3 功能命名

同一功能在文档、页面和服务中应使用统一名称。

例如：

```text
一键呼叫
候车热力图
座位二维码
下车唤醒
预计到站时间
共享电动车资格
线上考试
线下实操
```

不要在不同地方混用多个名称，例如“一键呼叫”“预约”“叫车”“呼车”混用。
