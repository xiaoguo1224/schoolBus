# 智慧通勤用户端与司机端微信小程序官方风格重构设计

## 1. 目标

本次设计面向 `wxapp/` 用户小程序与 `driver/` 司机小程序，目标是把当前偏演示稿、入口分散、视觉风格不统一的页面，重构为更贴近微信原生体验的两套独立小程序。

核心目标如下：

- 用户端与司机端仍保持两套代码，职责不混用
- 视觉与交互统一收敛到微信原生风格，避免 Web 化和重装饰
- 对现有功能做一次完整的信息架构重排，而不是只做换皮
- 把散落在首页和“我的”中的功能迁移到更合理的页面层级
- 保留当前已有功能范围，先重组结构，再逐步细化体验

## 2. 设计范围

本次只覆盖用户端和司机端的小程序页面结构、入口组织、页面职责、视觉风格和交互主线，不涉及后端接口、数据库、真实支付、真实地图 SDK 或权限体系实现。

范围内内容：

- `wxapp/` 用户小程序的信息架构重排
- `driver/` 司机小程序的信息架构重排
- 小程序首页、tabBar、详情页、列表页、个人中心的职责划分
- 现有功能在新结构中的迁移归属
- 微信原生风格的视觉与交互规则

范围外内容：

- 后端服务实现
- 真实接口联调
- 真实支付链路
- 生产级登录与权限控制
- 地图 SDK 深度接入
- 与管理后台 `fronted/` 的同步改造

## 3. 当前问题

当前两端页面的主要问题不是功能不足，而是功能堆放方式不清晰。

### 3.1 用户端的问题

- 首页同时承载班车、共享电动车、运营概览和快捷入口，主线不够明确
- `我的` 页面承担了太多职责，既像个人中心，也像功能总入口
- 线路、地图、扫码、权益、反馈等功能之间缺少层级关系
- 当前视觉风格偏演示页，装饰性强，和微信原生体验差距较大

### 3.2 司机端的问题

- 首页虽然已经是工作台，但任务、车辆、行程、异常、考勤和记录的边界仍然不够清楚
- 一些高频操作没有形成明确的主入口
- 页面之间的职责切分偏散，司机很难通过一条稳定路径完成“看任务 - 看车 - 跑行程 - 看记录”
- 视觉语言偏自定义工作台，不够原生、克制

## 4. 设计原则

### 4.1 微信原生优先

页面以微信小程序原生体验为准，不做 Web 化布局，不做复杂的装饰性大屏，不使用过重的玻璃拟态、强渐变或堆叠式营销卡片。

优先采用的表达方式：

- 原生导航栏
- 分组列表
- 简洁卡片
- 标签与状态徽标
- 单主按钮或底部操作栏
- 空状态、加载态、错误态

### 4.2 信息架构优先

本次重构的重点不是“把页面变好看”，而是先把功能放回正确的位置。

原则如下：

- 首页只保留最重要的场景入口和状态摘要
- 高风险、高频操作必须有独立入口或明确的二级入口
- 低频功能统一收纳到个人中心或记录中心
- 每个页面只承担一个清晰职责

### 4.3 两端独立

用户端和司机端继续保留两套代码，不合并为一个混合应用。

- `wxapp/` 只服务乘客侧
- `driver/` 只服务司机侧
- 两端可以共享设计语言，但不能共享业务职责

### 4.4 先收敛，再扩展

本次重构先把现有功能纳入清晰结构，后续再根据真实使用情况细分页面，不在本次设计里追求一步到位的超级大而全首页。

## 5. 用户端信息架构

用户端仍保持“班车通勤 + 共享电动车”双场景，但不再让首页同时承担太多解释性内容。

### 5.1 推荐入口结构

建议用户端保持 4 个底部入口：

1. 首页
2. 线路
3. 地图
4. 我的

其中“扫码乘车”“下车唤醒”“优惠券”“共享电动车资格”等功能不作为底部主入口，而是放在首页快捷操作、地图浮层入口或“我的”里的二级入口。

### 5.2 首页职责

首页是场景总览，不是所有功能的堆放页。

首页需要保留：

- 班车 / 共享电动车场景切换
- 当前场景的核心状态摘要
- 2 到 4 个最重要的快捷入口
- 今日运营或个人出行的关键提醒

首页不再承担：

- 过多的功能列表
- 过多的说明性文案
- 复杂的模块堆叠

### 5.3 线路页职责

线路页负责“看线路”，而不是混合所有出行功能。

线路页应覆盖：

- 线路列表
- 线路方向
- 站点顺序
- 当前运行状态
- 线路切换与高亮
- 线路详情入口

### 5.4 地图页职责

地图页负责“看车、看站、看 ETA、发起呼叫”。

地图页应覆盖：

- 车辆实时位置
- 车辆预计到站时间
- 站点分布与站点状态
- 一键呼叫
- 候车热度提示

### 5.5 我的页职责

“我的”不再是堆功能的杂物间，而是个人中心和服务中心。

我的页应覆盖：

- 个人信息
- 乘车记录
- 支付记录
- 退款申请
- 优惠券与乘车次数权益
- 问题反馈
- 司机评价
- 共享电动车资格入口
- 线上考试、线下实操、骑行记录入口

### 5.6 用户端页面归属

建议将现有与新增功能归入以下模块：

- 首页：双场景总览、快捷入口、关键提醒
- 线路：线路列表、线路切换、站点列表、线路详情
- 地图：车辆位置、ETA、候车热力、呼叫入口
- 扫码乘车：扫码、座位状态、支付确认
- 下车流程：下车点选择、下车唤醒、提醒设置
- 我的：记录、退款、券包、反馈、评价、共享电动车资格
- 共享电动车：车辆查看、线上考试、线下实操、资格、骑行记录

### 5.7 用户端现有页面迁移建议

| 现有页面 | 目标归属 | 说明 |
| --- | --- | --- |
| `pages/index/index` | 首页 | 双场景总览入口 |
| `pages/schedule/index` | 线路 | 线路与班次列表 |
| `pages/map/index` | 地图 | 实时车辆与 ETA |
| `pages/detail/index` | 线路详情 / 班次详情 | 从列表进入的详情页 |
| `pages/reserve/index` | 扫码乘车 | 座位状态、支付、下车流程 |
| `pages/mine/index` | 我的 | 个人中心与服务中心 |

### 5.8 用户端功能迁移原则

以下内容不建议继续堆在首页：

- 乘车记录
- 支付记录
- 退款申请
- 优惠券
- 乘车次数权益
- 问题反馈
- 司机评价
- 共享电动车资格与考试入口

这些功能应在“我的”里分组呈现，或者在首页只保留轻入口。

## 6. 司机端信息架构

司机端以“今日任务”为主线，围绕任务执行形成稳定的操作路径。

### 6.1 推荐入口结构

建议司机端保留 5 个底部入口，但重新定义职责：

1. 首页
2. 任务
3. 车辆
4. 行程
5. 记录

这样可以保留司机习惯中的多入口结构，同时让每个入口语义更稳定。

### 6.2 首页职责

首页是今日任务总览，不是装饰性看板。

首页需要保留：

- 今日任务摘要
- 当前在线状态
- 呼叫提醒摘要
- 热点站点摘要
- 下一步操作入口

首页的第一主线必须是“今天该跑什么、现在该处理什么”。

### 6.3 任务页职责

任务页负责当天任务的列表化管理。

任务页应覆盖：

- 今日任务列表
- 任务状态
- 候车呼叫提醒
- 热点站点提醒
- 任务详情入口

### 6.4 车辆页职责

车辆页负责车辆绑定与车辆状态确认。

车辆页应覆盖：

- 车辆绑定
- 车辆信息
- 座位数
- 车况检查
- 基础运行状态

### 6.5 行程页职责

行程页负责行程执行动作，是司机的核心操作页。

行程页应覆盖：

- 开始行程
- 到站确认
- 结束行程
- 异常上报入口
- 当前行程进度
- 乘客 / 座位摘要

### 6.6 记录页职责

记录页负责事后查看和复盘。

记录页应覆盖：

- 考勤记录
- 行程记录
- 异常记录
- 司机评价
- 历史提醒回执

### 6.7 司机端页面归属

建议将现有与新增功能归入以下模块：

- 首页：今日任务总览
- 任务：任务列表、呼叫提醒、热点提醒
- 车辆：车辆绑定、车辆状态、基础检查
- 行程：开始行程、到站确认、结束行程、异常上报入口
- 记录：考勤、评价、历史记录
- 个人信息、设置、登录状态等轻量内容放在首页或记录页的二级入口中，不单独作为底部主入口

### 6.8 司机端现有页面迁移建议

| 现有页面 | 目标归属 | 说明 |
| --- | --- | --- |
| `pages/index/index` | 首页 | 今日任务总览 |
| `pages/tasks/tasks` | 任务 | 今日任务、呼叫与热点 |
| `pages/vehicle/vehicle` | 车辆 | 车辆绑定与检查 |
| `pages/trip/trip` | 行程 | 开始、到站、结束、异常入口 |
| `pages/report/report` | 行程 / 任务 | 异常上报，不建议独立成为主入口 |
| `pages/attendance/attendance` | 记录 | 考勤与评价 |
| `pages/logs/logs` | 记录 | 历史记录与回执 |

### 6.9 司机端功能迁移原则

以下内容不应继续分散在首页：

- 车辆绑定细节
- 异常上报表单
- 考勤明细
- 历史流水
- 评价记录

它们应各自归到任务、车辆、行程或记录中。

## 7. 视觉风格

### 7.1 用户端风格

用户端采用轻、干净、克制的微信原生风格。

建议特征：

- 白底或极浅灰底
- 少量绿色或青绿色作为主强调色
- 以列表、卡片、标签和按钮为主
- 以自然留白和信息层级来建立秩序
- 不使用大面积强渐变和装饰型背景

### 7.2 司机端风格

司机端同样保持原生，但视觉上更偏工作台和运营工具。

建议特征：

- 白底为主，辅以蓝色强调
- 状态感更明确，信息密度略高于用户端
- 强调“当前任务”“下一步动作”“提醒状态”
- 页面更像工作台，而不是营销页

### 7.3 共同规范

两端都应避免：

- 玻璃拟态
- 复杂背景纹理
- 大量无意义图标堆叠
- Web 风格 hover 或复杂层叠
- 用“展示页”替代“工作流页面”

## 8. 交互规则

### 8.1 用户端主流程

用户端最重要的主流程应清晰可达：

1. 首页选择班车或共享电动车场景
2. 进入线路或地图查看状态
3. 需要乘车时进入扫码乘车
4. 支付后选择下车点并开启唤醒
5. 后续在“我的”里查看记录、退款和权益

### 8.2 司机端主流程

司机端最重要的主流程应清晰可达：

1. 打开首页看到今日任务
2. 进入任务或车辆页完成准备
3. 在行程页执行开始、到站、结束
4. 遇到问题时进入异常上报
5. 结束后在记录页查看考勤和历史

### 8.3 状态展示

所有页面的状态反馈都要贴近微信原生习惯：

- `loading` 用轻量加载态
- `empty` 用简洁空状态
- `success` 用明确的结果反馈
- `warning` 用标签、提示条或轻弹窗
- `error` 用可理解的错误说明

## 9. 迁移策略

### 9.1 保留两套代码

本次不改变 `wxapp/` 与 `driver/` 的独立代码结构。

### 9.2 先重排，再细化

建议先完成以下工作：

- 统一页面骨架和导航
- 重排入口和职责
- 把现有功能放入新结构
- 再逐页调整视觉和交互细节

### 9.3 现有内容处理方式

现有页面中的内容不要求全部删除，而是按新职责迁移：

- 能保留的功能数据继续保留
- 功能归属不清的内容优先迁到“我的”“记录”或二级详情页
- 首页只保留最必要的入口和摘要

### 9.4 迁移优先级

优先级建议如下：

1. 首页与底部导航
2. 线路 / 地图 / 我的
3. 今日任务 / 任务 / 车辆 / 行程 / 记录
4. 扫码乘车与下车流程
5. 优惠券、反馈、评价、共享电动车资格相关页

## 10. 验收标准

完成后至少应满足以下条件：

- 用户端和司机端都具有清晰的一条主线
- 首页不再承担所有功能堆叠
- 用户端可以在合理层级找到线路、地图、扫码和个人服务
- 司机端可以在首页快速看到今日任务并进入执行流程
- 页面视觉统一为微信原生风格，不再像 Web 演示页
- 所有现有功能都有明确归属，不存在“无家可归”的功能

## 11. 非目标

本次不做以下内容：

- 真正的业务接口实现
- 真正的支付接入
- 真正的地图定位优化
- 复杂动画和炫技式设计
- 管理后台 `fronted/` 的同步重构
- 后端 `backend/` 的功能改造

## 12. 结论

本次重构的核心不是“变得更花”，而是“变得更像微信小程序，也更像一个能长期维护的业务系统”。

用户端要从“什么都有一点”变成“场景清楚、入口清楚、服务清楚”。

司机端要从“多个页面并列”变成“今日任务主导、执行流程明确、记录可追溯”。
