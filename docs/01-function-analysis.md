# 智慧通勤综合服务平台功能分析（Codex 开发版）

> 本文档面向 Codex 和开发人员，用于明确“要开发什么”。  
> 本文档只保留可开发、可拆分、可验收的功能范围。  
> 详细汇报说明、界面截图、图片提示词、长篇背景说明应放到汇报文档，不放在本文档中。

---

## 1. 项目定位

本项目是面向校园 / 园区通勤场景的智慧通勤综合服务平台。

系统包含三类主要端：

```text
用户微信小程序
司机微信小程序
管理后台 Web
```

系统核心能力：

```text
班车出行
实时车辆位置
ETA 预计到站
一键呼叫
扫码乘车
座位二维码
订单支付
下车地点选择
到站提醒
优惠券
乘车次数权益
问题反馈
司机评价
共享电动车入口
共享电动车资格认证
后台运营调度
```

系统不以“用户提前预约班车”为核心，而是以“实际出行过程”为核心：

```text
查看线路 / 站点 / 车辆
到站点后一键呼叫
上车后扫码乘车
支付后选择下车点
接收到站提醒
后台根据实时数据调度
司机接收任务和提醒
```

```text
到站确认
结束行程
扫码乘车

支付记录
乘车记录
下车点选择
到站提醒
管理后台基础数据管理
管理后台地图调度看板
管理后台订单管理
管理后台二维码管理
管理后台一键呼叫管理
管理后台候车热力图
共享电动车资格状态
线上考试
线下实操审核
问题反馈
司机评价
```

### 3.2 第一版可简化实现

```text
真实微信支付：第一版可用模拟支付，后期接微信支付
ETA：第一版粗略估算即可
车辆轨迹：第一版可按 30～60 秒采样入库
热力图：第一版按站点候车人数生成
优惠券：第一版支持基础领取与抵扣
乘车次数权益：第一版支持简单满次赠送
共享电动车：第一版只做入口、车辆展示和资格认证，不做硬件控制
后台统计：第一版做基础统计，不做复杂 BI
包车服务：第一版只做申请记录，不做自动派车和支付闭环
紧急呼叫：第一版只做保卫处电话展示与拨打
```

### 3.3 第一版不做

```text
不做普通班车预约
不做复杂拼车预约
不做完整共享电动车硬件控制
不做复杂公交级精准 ETA
不做大数据实时计算平台
不做多端 App
不做微服务拆分
不做 Kafka 流处理
```

### 4.7 一键呼叫

一键呼叫字段：

```text
用户

线路
起始站点
目标站点
用户定位
呼叫时间
呼叫状态
处理状态
```

业务规则：

```text
起始点通过用户定位确定
用户必须选择目的地站点
用户必须在站点候车范围内才能呼叫
同一用户同一站点短时间内不能重复呼叫
呼叫记录必须写入 PostgreSQL
当前候车人数必须写入 Redis
呼叫成功后发布 call.created 消息
司机端和后台应收到呼叫提醒
```

验收标准：

```text
用户可以在站点附近发起呼叫
用户不在站点范围内不能呼叫
用户重复呼叫会被拦截
呼叫记录可以在后台查看
站点候车人数实时增加
司机端可以收到呼叫提醒
后台调度看板可以看到呼叫点
```

### 4.8 候车人数统计

数据来源：

```text
一键呼叫
扫码乘车
用户定位
历史乘车记录
后台人工录入
```

业务规则：

```text
当前候车人数以 Redis 为准
历史统计以 PostgreSQL 汇总为准
禁止用数据库 count(*) 实时计算当前候车人数
```

验收标准：

```text
用户可以看到站点候车人数
司机可以看到呼叫站点候车人数
后台可以看到站点候车人数和热力图
```

### 4.9 扫码乘车

用户上车后扫描座位二维码，进入乘车确认与支付页面。

页面展示：

```text
车辆
座位
线路
行程
上车站点
支付金额
车辆座位总数
空座数
已占座数
当前座位状态
```

业务规则：

```text
每个座位一个二维码
扫码后必须校验二维码、车辆、行程和座位状态
座位状态需要用颜色区分
座位编号必须清晰展示
扫码后不能直接认为支付成功
```

座位颜色：

```text
绿色：可用
红色：已占用
灰色：不可用
黄色：锁定中
```

验收标准：

```text
用户扫码后可以看到车辆和座位信息
用户可以看到座位状态图
不可用或已占用座位不能继续支付
有效座位可以创建订单
```

## 9. 核心业务流程

### 9.1 车辆实时位置流程

```text
司机开始行程
        |
司机端周期性上报位置
        |
后端校验司机、车辆、行程
        |
写 Redis 最新车辆状态
        |
发布 vehicle.location.updated
        |
ETA 异步计算
        |
WebSocket 推送用户端、司机端、后台
        |
轨迹按 30～60 秒采样入库
```

### 9.2 一键呼叫流程

```text
用户选择目的地站点
        |
用户点击一键呼叫
        |
系统获取用户定位
        |
校验是否在站点候车范围
        |
校验 Redis 防重复呼叫
        |
写 call_records
        |
Redis 增加站点候车人数
        |
发布 call.created
        |
司机端和后台收到提醒
```

### 9.3 扫码乘车流程

```text
用户扫描座位二维码
        |
后端校验二维码、车辆、行程、座位
        |
Redis / Redisson 加座位锁
        |
创建订单
        |
模拟支付
        |
更新订单和支付记录
        |
更新座位状态
        |
生成乘车记录
        |
选择下车站点
        |
开启到站提醒
```

### 9.4 到站提醒流程

```text
车辆位置更新
        |
ETA 计算
        |
用户已选择下车站点
        |
ETA 小于阈值 / 车辆进入站点范围 / 司机到站确认
        |
触发 arrival.reminder
        |
用户收到小程序消息、页面弹窗或震动提醒
```

## 11. Codex 开发硬约束

Codex 开发时必须遵守：

```text
不做普通班车预约。
一键呼叫必须写数据库留痕。
一键呼叫当前候车人数必须使用 Redis。
禁止用数据库 count 实时计算候车人数。
车辆实时位置必须写 Redis。
车辆实时位置禁止每次都写数据库。
车辆轨迹只允许采样入库。
扫码乘车必须校验二维码、车辆、行程、座位。
扫码乘车必须使用 Redis / Redisson 座位锁。
订单状态必须使用状态机。
座位状态必须使用状态机。
支付回调必须幂等。
优惠券核销必须防重复。
所有核心数据必须带 campus_id。
所有后台查询必须按 campus_id 隔离。
司机只能访问自己的任务。
用户只能访问自己的订单和记录。
管理员只能访问本校园数据。
WebSocket 只负责推送，不作为最终状态来源。
RabbitMQ 消费者必须幂等。
文件上传必须进入 MinIO，不直接存数据库。
```
